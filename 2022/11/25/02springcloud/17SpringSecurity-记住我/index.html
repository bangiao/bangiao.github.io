RememberMe

基础

是什么?(为什么?)

Remember Me 即记住我，常用于 Web 应用的登录页目的是让用户选择是否记住用户的登录状态。当用户选择了 Remember Me 选项，则在有效期内若用户重新访问同一个 Web 应用，那么用户可以直接登录到系统中，而无需重新执行登录操作。

Remember Me是一种服务器行为, 由服务器记录信息

怎么做?(quick start)

以后用户拿着如上图所示的remember-me的值跟服务器进行用户身份校验

小白: “直接暴露了我们的remember me的配置，也就是series和token。存在安全隐患吧?”

小黑: “是的, 这样黑客就可以根据他截获的cookie中remember me的值进行伪造访问，直接获取到我们数据库中个人信息。”

小黑: “如果我们仔细查看TokenBasedRememberMeServices的话，就会发现他每一次的remember-me的值都不会改变。这样更加不安全.”

小黑: “真的要买就4行核心代码。第1部分主要是拿到过期时间，第2部分拿到内存中的用户信息，第3部分就是创建出一个签名，最后对比一下完事。所以这样子的话，最后客户端cookie的值还是不会发生任何的改变。”

但是spring security也提供了一些解决方案。我们只要进自定义的TokenRepository, 我们就会走PersistentTokenBasedRememberMeServices, 这里的token会不断发生变化, 这样虽然还是能够被窃取一次, 但总体安全, 我们还能够配合_csrf进行验证,基本上完美解决窃取问题

持久化令牌

将令牌从内存转移到数据库

小黑: “这种方式是换了一种策略去实现remember-me功能。使用TokenBasedRememberMeServices变成使用PersistentTokenBasedRememberMeServices“

小白: “那我们要怎么去切换这种策略呢？”

小黑: “非常简单，存在两种方式。”

小白: “两相对比一下，我喜欢第二种方式”

持久化令牌跟数据库相关, 与传统方式相比较, 多出了两个key: series和token.

series仅在用户使用密码重新登录时更新，而 token 会在每一个新的session会话中都重新生成

持久化令牌方案 避免了散列加密方案中，一个令牌可以同时在多端登录的问题，这是因为每个session会话都会引发token的更新，即每个token仅支持单实例登录。

自动登录不会导致series变更，但每次自动登录都需要同时验证 series和 token两个值，所以这样的设计会更安全

因为当该令牌还未使用过自动登录就被盗取时，系统会在非法用户验证通过后刷新 token 值，此时在合法用户的浏览器中，该token值已经失效。当合法用户使用自动登录时，由于该series对应的 token 不同，系统可以推断该令牌可能已被盗用，从而做一些处理。例如，清理该用户的所有自动登录令牌，并通知该用户可能已被盗号等。

底层都做了什么?

小白方案

小黑: “小白我考考你, 如果让你设计rememberMe功能，你要怎么设计？”

小白: “让我设计的话我会将 session中的内容读写出来存放到数据库中或者是第三方缓存中，这样即使微服务重启或者用户关闭浏览器等, 就会第一时间读取到这些数据。”

小白: “所以我需要在session中没有我们的用户信息时，先判断一下我们存储的数据库是否存在用户信息，然后再考虑是否实施用户登录行为。”

小黑: “这我就有话说了, 服务端的session过期了, 那你怎么查到我们数据库中的数据呢?”

小白: “我想想, 我觉得可以存储到 cookie 中, 但时候先解析cookie看看能不能拿到相关数据先”

小黑: 你说的流程大体都对，但实际上还是有一些问题的。

  • 第1点，你的session的内容太多了，其实只需要存储一些必须的东西，就是比如说用户名, 创建时间等。如果你session中的所有内容都拷贝并保存在数据库的话，会导致资源的浪费，毕竟用户很多。

  • 第2点, 我们对存储的数据也是有要求的，存储的数据必须保证它是唯一的。存储的数据还需要一些日期。因为可能会有同名用户多次登录的情况。而且是在不同的设备上。

  • 第3点，如果用户在跨设备上使用的话，那么那么你这台设备就必须要有一个唯一匹配的数据。比如你的账户小明在a电脑上登录了，现在你去网吧，电脑上也登录了小明这个用户，那你怎么判断是在电脑A上登录还是在网吧电脑上登录的呢？所以在电脑上还是需要存放一些唯一标识的，这就跟上面第2点对上了。

二次校验功能的实现

很多敏感URL，需要二次输入密码认证。所以便拥有了本章的内容。

先给出规则:

  • authenticated方法: 不论是账号密码方式登录还是rememberMe方式登录都可以访问url
  • .antMatchers("remember").rememberMe()方法: 只有rememberMe方式认证完毕后才能访问url, 其他方式都不可以
  • .antMatchers("/again").fullyAuthenticated()方法: 只能使用账户密码方式登录才能访问url, rememberMe方式访问不行

只要记住:

通用

只能rememberMe

只能账户密码

现在我们以密码登录和记住我登录两种方式登录我们的网页

密码登录比记住我登录安全, 所以我们的 url 也分为两种类型, 敏感型和非敏感型

敏感型需要输入密码登录, 非敏感型 两种方式都能够登录

现在就能够有这种情况, 我们配置一个只能够通过rememberMe访问的url: /remember

另外再配置一个只能够使用账号密码方式访问的url: /auth

还有一个两种方式都能访问的url: /both

这里注意, 登录页面输入账号密码登录就是fullyAuthenticated模式登录, 即便你选择了rememberMe按钮记住我登录也算fullyAuthenticated方式登录

实战

配置完成之后, 你会发现不管你在登录的时候是否点击了 rememberMe 按钮, /remember url 都无法访问, 都是没有权限, 因为他们都是账号密码方式登录, 而非rememberMe 方式登录, 直到你关闭浏览器, 再打开浏览器, 此时访问我们的remember url你会发现它可以直接登录, 而auth url 不能访问了, 大意是关闭浏览器再打开就创建了一个新的会话, 此时就会执行一次自动登录, 也就是rememberMe方式登录, 此时被配置为fullyAuthenticated的/auth就无法访问了

根据上面的提示, 我们在实战中应该保护的是/auth url, 所以/remember url 就不要配置 rememberMe了, 直接删除掉

意思是说, 只要是需要保护的 url 全部挂上fullyAuthenticated, 只有使用密码登录才能访问

现在三个 url 都可以访问

此时关闭浏览器, 再打开浏览器, 此时再访问/auth 将自动跳转到 登录页面
