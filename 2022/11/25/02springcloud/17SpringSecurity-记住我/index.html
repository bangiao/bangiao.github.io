17SpringSecurity-记住我
2022-11-25|springcloud

RememberMe

基础

是什么?(为什么?)

Remember Me 即记住我，常用于 Web 应用的登录页目的是让用户选择是否记住用户的登录状态。当用户选择了 Remember Me 选项，则在有效期内若用户重新访问同一个 Web 应用，那么用户可以直接登录到系统中，而无需重新执行登录操作。

Remember Me是一种服务器行为, 由服务器记录信息

小白方案

小黑: “小白我考考你, 如果让你设计rememberMe功能，你要怎么设计？”

小白: “让我设计的话我会将 session中的内容读写出来存放到数据库中或者是第三方缓存中，这样即使微服务重启或者用户关闭浏览器等, 就会第一时间读取到这些数据。”

小白: “所以我需要在session中没有我们的用户信息时，先判断一下我们存储的数据库是否存在用户信息，然后再考虑是否实施用户登录行为。”

小黑: “这我就有话说了”

初次登录

小黑: 你说的流程大体都对，但实际上还是有一些问题的。

  • 第1点，你的session的内容太多了，其实只需要存储一些必须的东西，就是比如说用户名, 用户状态等。如果你session中的所有内容都拷贝并保存在数据库的话，会导致资源的浪费，毕竟用户很多。

  • 第2点, 我们对存储的数据也是有要求的，存储的数据必须保证它是唯一的。存储的数据还需要一些日期。因为可能会有同名用户多次登录的情况。而且是在不同的设备上。

  • 第3点，如果用户在跨设备上使用的话，那么那么你这台设备就必须要有一个唯一匹配的数据。比如你的账户小明在a电脑上登录了，现在你去网吧，电脑上也登录了小明这个用户，那你怎么判断是在电脑A上登录还是在网吧电脑上登录的呢？所以在电脑上还是需要存放一些唯一标识的，这就跟上面第2点对上了。

小白: “那spring security要怎么实现这些呢？”

小黑: “首先spring security要有一个配置，这个配置去启动，remember-me操作。”

小黑: “这样前端就可以在表单上加上一个key value操作。key = 'remember-me', value = 'on/off'“

小黑: “后端接收到这个表单之后。去判断前端用户是否勾选了remember me操作。收到后端，收到前端的remember-me操作之后，将会创建两个属性: series和token“

小黑: “上面的两个generate函数就是生成的两个那两个属性。里面的实践方法其实很简单，就是两个UUID, 然后通过 Base64 加密了下”

小白: “那为什么要配置两个这样的属性呢？有什么作用？”

小黑: “token表示一个会话。 series是表示用户用用户名和密码登录。这样设计的好处是可以多电脑登录。”

小白: “那你继续讲。”

小黑: “后端会将生成的两个属性和用户名，还有最后登录的时间一起保存到数据库中。”

小黑: “但是需要注意的是，本次登录其实使用的是用你密码方式登录，而不是remember me方式登录，所以存在一个问题，就是后面如果有一些URL，它配置为只有remember me才能访问的话”

小黑: “因为我们使用的是用户名密码登录，所以无法访问。如果浏览器被关闭之后，然后再次打开此时，在访问remember me的URL才可以访问，因为用户关闭浏览器和重新打开浏览器，这一过程其实是会话的关闭和开启。此时才算remember-me方式登录。”

如何维持remember-me登录?

小白: “那现在有个问题了，你竟然把数据保存在了数据库中，那我下次把会话关闭，然后重新打开。怎么判断我还是自动登录呢？”

小黑: “你要注意一点，spring security它是一个过滤器集合，也就是它是管理一个过滤器的。所以记住我这项功能，它也是一个过滤器。所以我们只要保证这个过滤器，跳转到自动登录页面之前，判断一下到底是不是自动登录方式登录的系统就可以了。”

小白: “具体要怎么操作呢？”

小黑: “你要记住，每一个浏览器在访问。服务器的时候都会有cookie和session, 如果你是以记住我方式，登录过我们的系统，那么你在cookie中就一定会保存相关的标识。”

小黑: “ Cookie里面存储的是两个属性的加密。中间使用:进行分割。”

小黑: “此时你只需要将cookie中存储的信息发送给后端，后端会解析出相关的属性，这两个属性就是前面我们已经讲过的token和series是属性。现在拿着这两个属性去数据库中读取相关的自动登录信息。如果从数据库中读取到相关的信息。那么它就不会跳转到登录页面，而是继续访问我们的网页。”

小白: “等等, 那前面讲过的spring security上下文SecurityContextHolder呢？他应该是保存在session中吧。 Session, 噢，对噢，它保存在session中。没事了没事了”

小黑: “你这么说的话那是错的，其实他在启动的时候会去数据库中读取你保存的信息。”

小黑: “读写完之后他会带着上面的username去读取users表。”

小黑: “上面还做了一些判断，首先你拿出了这些数据的时候，他会首先判断token，等判断完token之后再继续判断是否过期，这个时间定在两周，如果超过两周它就过期了。”

小黑: “在最后的返回地方就拿着username去user表里面去查。等到查询出来之后呢，存储到新创建的SecurityContextHolder上下文。”

小白: “哦，那微服务重启的话也是直接从这里拿了，所以 Spring security的上下文其实是没什么问题的。”

小黑: “对的情况就是这么个情况。这里还有一个地方值得关注。”

小黑: “看到没token更新了，但是series没更新。这也印证了前面的那个那句话，token会跟随着绘画的创建而创建，所以每次有一个新的会话的话，那么就会有一个新的token，但是series只会有一个。除非你用用户名和密码重新登录了一次。”

总结

小白: “那我最后再做一个总结吧。记住我操作的话，他需要spring security手动开启，开启之后前端还要加上key value为remember me的存在。如果前端remember me的值为on, 他就会将这些数据存储在数据库(存储在数据库的过程，需要程序员去自定义实现它。)。这是第1次登录时勾选remember me的过程。”

小白: “如果会话被销毁。也就是说浏览器被关闭。那么在下次打开浏览器访问我们的网站时。 Remember me的过滤器会拦截请求。并从数据库中读取先前保存的remember me对象的数据。如果该数据不存在那么走重新认证的过程，也就是跳转的登录页面让用户输入用名和密码重新登录。如果从数据库中查询到remember me对象的话。那么将从中创建一个新的token属性。并且在将该对象的username内读取出来。用于查询数据库中的users对象。将该查询出来的对象存放到一个新创建的SecurityContextHolder上下文。完毕。”

持久化令牌

搞得好像很高级的样子，其实就是把我们前面的PersistentRememberMeToken对象存储到数据库中而已。

持久化令牌跟数据库相关, 与传统方式相比较, 多出了两个key: series和token.

series仅在用户使用密码重新登录时更新，而 token 会在每一个新的session会话中都重新生成

持久化令牌方案 避免了散列加密方案中，一个令牌可以同时在多端登录的问题，这是因为每个session会话都会引发token的更新，即每个token仅支持单实例登录。

自动登录不会导致series变更，但每次自动登录都需要同时验证 series和 token两个值，所以这样的设计会更安全

因为当该令牌还未使用过自动登录就被盗取时，系统会在非法用户验证通过后刷新 token 值，此时在合法用户的浏览器中，该token值已经失效。当合法用户使用自动登录时，由于该series对应的 token 不同，系统可以推断该令牌可能已被盗用，从而做一些处理。例如，清理该用户的所有自动登录令牌，并通知该用户可能已被盗号等。

二次校验功能的实现

先给出规则:

  • authenticated方法: 不论是账号密码方式登录还是rememberMe方式登录都可以访问url
  • .antMatchers("remember").rememberMe()方法: 只有rememberMe方式认证完毕后才能访问url, 其他方式都不可以
  • .antMatchers("again").fullyAuthenticated()方法: 只能使用账户密码方式登录才能访问url, rememberMe方式访问不行

只要记住:

通用

只能rememberMe

只能账户密码

现在我们以密码登录和记住我登录两种方式登录我们的网页

密码登录比记住我登录安全, 所以我们的 url 也分为两种类型, 敏感型和非敏感型

敏感型需要输入密码登录, 非敏感型 两种方式都能够登录

现在就能够有这种情况, 我们配置一个只能够通过rememberMe访问的url: /remember

另外再配置一个只能够使用账号密码方式访问的url: /auth

还有一个两种方式都能访问的url: /both

这里注意, 登录页面输入账号密码登录就是fullyAuthenticated模式登录, 即便你选择了rememberMe按钮记住我登录也算fullyAuthenticated方式登录

实战

配置完成之后, 你会发现不管你在登录的时候是否点击了 rememberMe 按钮, /remember url 都无法访问, 都是没有权限, 因为他们都是账号密码方式登录, 而非rememberMe 方式登录, 直到你关闭浏览器, 再打开浏览器, 此时访问我们的remember url你会发现它可以直接登录, 而auth url 不能访问了, 大意是关闭浏览器再打开就创建了一个新的会话, 此时就会执行一次自动登录, 也就是rememberMe方式登录, 此时被配置为fullyAuthenticated的/auth就无法访问了

根据上面的提示, 我们在实战中应该保护的是/auth url, 所以/remember url 就不要配置 rememberMe了, 直接删除掉

意思是说, 只要是需要保护的 url 全部挂上fullyAuthenticated, 只有使用密码登录才能访问

现在三个 url 都可以访问

此时关闭浏览器, 再打开浏览器, 此时再访问/auth 将自动跳转到 登录页面
