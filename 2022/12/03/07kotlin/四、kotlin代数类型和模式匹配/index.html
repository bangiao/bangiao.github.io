代数数据类型

本章使用密封类、数据类去构建一个代数数据类型

代数数据类型

代数数据类型英文名叫 Algebraic Data Type(ADT)

本章主要内容:

  1. 什么是代数?
  2. 什么是代数数据类型?
  3. 为什么需要代数数据类型?
  4. 怎么使用代数数据类型?

什么是代数?

代数 分为 代 (功能) 数(目标)

数字在小学数学书上就有: 0, 1, 2, ... 100 这些就是数

带 代 是什么?

代表示代替, 组合在一起就是 代替数字 的 符号

比如我们初中学的 一元一次方程, x + 5 = 6

这里的 x 就是代数, 而代数的值是 1

y * 3 = 21 中的 y 也是一个代数, y 的值是 7

什么是代数数据类型?

请注意: 前面的方程中还有两个操作符, + *

我们可以还可以通过代数和数字组合成新的代数:

x * 1 = z

a + 2 = c

如果我们将上面的两个方程式改编成程序的表达式, 就会发现代数 z 的类型是另一个 代数 + 数字 的组合, 我们将这种组合归纳成一种新的类型, 这种类型被叫做 代数数据类型(ADT), 而在程序角度看, z 就是代数数据类型

为什么需要代数数据类型?

我们可以配合 “操作符” + “基本数据类型” 抽象成比较复杂而且功能强大的类型, 以完成更加复杂的业务需求

并且代数数据类型比较安全, 可以免去很多的麻烦(这个后面会提到)

我们的枚举类也是一种代数数据类型

前面介绍的代数都是一些初等代数, 代数是一个非常庞大的数学分支, 从简单的线性, 多项式代数环 域 再到 范畴 函子等, 更加抽象的代数

往后发展 代数将会变得更加复杂, 抽象级别越高, 也越发接近事物的本质

以外, 在日常开发中, 如果能够合理地利用ADT去对业务进行高度抽象, 那么我们的代码在能够实现诸多功能的前提下, 还会变得非常简洁.

计数

通过计数, 我们可以更加好的理解ADT.

什么是计数?

每种类型在实例化的时候, 都会有对应的取值, 比如: Boolean 类型存在两种可能的取值: true 和 false. 如果我们就将数字2与Boolean的取值种类相关联, 这种方式就叫做 计数

可能还不好理解, 再给个例子吧

我们知道 kotlin 有一个 类型 叫 Unit , 那么 Unit 的计数对应什么数字呢?

我们知晓, Unit 只能表示一个实例, 也就是说, 它只有一种取值, 所以采用计数的方式, 所以 Unit 对应数字 1

有了计数之后, 我们可以更加好的理解什么是 积类型(product) 与 和类型(sum)

和类型与密封类

前面说的积类型对应着前面的 乘法, 那么和类型顾名思义对应着代数中的加法.

有个问题需要判断, 那就是枚举是积类型还是和类型?

这个很好判断:

我们可以计算他的计数

它总共有这么几种结果:

1 + 1 + 1 + 1 + 1 + 1 + 1 = 7

它总共有 7 种取值的可能

我们通过计算 计数 的过程使用的是 + 操作符, 所以它是 和类型

我们来总结下和类型(sum) 的特点:

  1. 和类型是类型安全的, 它是一个闭环, 我们知晓他们的所有取值可能, 除了这些取值可能外, 就没有其他值了, 不用去担心非法情况
  2. 和类型是一种 OR关系. 作为比较, 积类型可以拥有好几种类型是一种 and 关系. 比如前面的 BooleanProductUnit就是同时拥有Boolean 和 Unit, 这种同时就是 AND 关系的一种表现. 而和类型一次只能是其中的某种类型, 要么 是 SUN 要么是 MON, 不能同时拥有两种类型, 所以它代表 OR 的含义

在 kotlin 1.1 之后, 密封类中 的 SUN 这些子类可以不用写在 Day 类作用域内, 可以写在同一个文件中

密封类的好处是配合 when, 使用 when 可以不用再添加 else 分支了

只要你添加了新的分支, 就会报错, 如果你没有添加完整的分支, 也会报错

构造代数数据类型(怎么用?)

我们现在需要判断一个几何图形是 三角形? 圆形? 还是长方形?

拿到这个问题域, 我们可以知晓, 这几个几何图形太过具体, 需要进行抽象处理

提取可抽象的部分: 图形, 定义密封类 Shape

这样我们就定义出一个 ADT 和类型: Shape 和 Circle Rectangle 和 Triangle 三个积类型

Shape 是和类型, 很好判断: 因为他的计数通过 + 加号计算出来的

Circle Rectangle 和 Triangle 三个是积类型也很好判断: 通过 Double 类型组合成积类型

这样我们就可以使用 密封类 配和 when表达式

ADT需要配合模式匹配才能够体现出最大的作用, 但是 kotlin 并没有在几大的程度上支持模式匹配, 然而 kotlin 表达式一九四非常强大的语言特性

模式匹配的使用场景

遇到一些复杂的数据结构, 比如树. 在对树进行操作的时候, 我们可以访问其结构内部的某个属性, Java 常用的方法是定义一个 类似 getter 的函数, 来访问某个属性

然而这显然不现实, 那些复杂的数据结构并没有给我们提供那么多方法. 有时候我们可能很难可以直接定位到我们需要的元素, 而是一层一层的访问过去, 然后再获取其中的属性. 进行相关操作, 比如 链表.

增强Kotlin模式匹配

实现模式匹配的技术都有:

  1. 类型测试/类型转换(Type-Test/Type-Cast)
  2. 面向对象分解(Object-Oriented Decomposition)
  3. 访问者设计模式(Visitor)
  4. Typecase
  5. 样本类(Case Classes)
  6. 抽取器(Extractor)

只有后面3种 Kotlin 还不能实现, 或者说实现起来比较困难

类型测试/类型转换

Kotlin可以进行类型测试和Smart Casts 智能类型转化

但是这种方式会存在一些缺点, 所以这种方式基本无法增强koltin的模式匹配能力
